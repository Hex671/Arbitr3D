## 二、 核心数据结构设计 (Core Data Structures)
在动手写核心逻辑前，明确模块间传递的数据结构至关重要。建议在根目录建立一个 `datatypes.py` 或在各模块中遵循以下字典/对象格式：

1. **渲染索引图 (Index Map / Z-buffer)**
   - **Shape**: `(H, W, 1)`
   - **含义**: `index_map[y, x]` 存储的是该像素对应原始 3D 点云数组中的真实 `point_index`。背景可设为 `-1`。
2. **3D 实体提议 (Entity3D)**
   - **定义**: 经过阶段 2 融合与隔离后的绝对物理部件。
   - **属性**:
     - `entity_id`: 唯一标识符 (int，如 1, 2, 3...)。
     - `point_indices`: 该实体包含的 3D 点云索引列表 `List[int]`。
     - `glip_scores`: 收集到的原始类别置信度字典 `Dict[str, float]`。

## 三、 分阶段模块详细设计 (Stage-by-Stage Implementation)

### 【阶段 0】数据准备与配置
*   **`config/config.yaml`**: 定义全局变量（摄像机数量 $K=20$、合并阈值 $IoU=0.3$、SAM 权重路径等）。
*   **`data_prep/dataloader.py`** -> `PointCloudDataset` 类：
    *   `normalize_pc(points)`: 点云归一化，平移至原点并缩放至单位球内，确保 `module_render` 的机位能完美包裹对象。

### 【阶段 1】多视角渲染与 2D 基础模型爆发式提取
*   **`module_render/camera_poses.py`**:
    *   `generate_sphere_cameras(K=20)`: 基于斐波那契网格生成均匀分布的摄像机外参。
*   **`module_render/renderer.py`** -> `MultiViewRenderer` 类:
    *   **输入**: 归一化点云。
    *   **输出**: `images` (K张 RGB), `index_maps` (K个形状为 HxWx1 的索引矩阵)。
*   **`module_2d_fm/glip_extractor.py`**:
    *   `get_bboxes_and_scores(images, prompt_classes)`: 批量调用 GLIP，返回每张图的 BBox 列表和候选类别分数。
*   **`module_2d_fm/sam_segmenter.py`**:
    *   `generate_masks_from_bboxes(images, bboxes)`: 将 BBox 作为 prompt 喂给 SAM，输出高精度的 2D 布尔掩码矩阵 (`masks`)。

### 【阶段 2】3D 空间反投影、融合与硬隔离 (Arbitr3D 工程核心)
*   **`module_3d_lift/projector.py`**:
    *   `back_project_2d_to_3d(masks, index_maps)`: 遍历 `masks` 中值为 True 的像素，从 `index_maps` 中提取对应的 3D 点索引，生成初始的“局部点簇”列表。
*   **`module_3d_lift/merger.py`**:
    *   `calculate_3d_iou(cluster_a, cluster_b)`: 计算两个点簇的 3D 交并比。
    *   `macro_merge(clusters, threshold=0.3)`: **宏观合并**。按包含点数降序排列，计算重叠度，将属于同一物理部件的点簇合并为初始的 `Entity3D`，并把 GLIP 分数汇总。
*   **`module_3d_lift/conflict_resolver.py`** (💡 解决过度分割与重叠的杀手锏):
    *   `resolve_overlaps(entities)`: **微观硬隔离**。找出所有实体之间共享的 `point_index`。执行**细粒度优先原则**：如果一个点同时属于大体积实体 $E_A$ 和小体积实体 $E_B$，强制从 $E_A$ 中剔除该点，判给 $E_B$。
    *   `graph_cut_smoothing(entities, point_cloud)`: 基于 3D KNN 建立邻接图，执行多数投票，消除因为投影导致的边界锯齿和“飞地”。
    *   **输出**: $M$ 个完美互斥的 `Entity3D`。

### 【阶段 3】MLLM 先验压缩与关键视角生成
*   **`module_mllm/prior_compressor.py`**:
    *   `aggregate_glip_scores(entities)`: 累加并归一化每个 `Entity3D` 的 `glip_scores`，生成如 `{"ID 1": {"椅背": 0.8, "座垫": 0.2}, "ID 2": ...}` 的浓缩先验字典。
*   **`module_mllm/key_view_selector.py`**:
    *   `greedy_view_selection(entities, index_maps)`: 遍历 $K$ 个视角的 `index_maps`，统计每个视角能看到的实体数量和面积。使用贪心算法选出 4~6 个能完整、无遮挡地展现这 $M$ 个实体的**关键机位索引**。

### 【阶段 4】MLLM 全局逻辑仲裁
*   **`module_mllm/som_renderer.py`**:
    *   `render_som_collage(images, entities, key_views)`: 取出关键机位的 RGB 图，根据 $M$ 个实体反向在 2D 图上覆盖高对比度半透明遮罩，在中心绘制带白边的显眼数字标签（如 `[1]`, `[2]`）。将 4~6 张图标注后拼贴成一张大图。
*   **`module_mllm/mllm_arbitrator.py`**:
    *   `construct_prompt(prior_dict)`: 构造系统提示词，注入先验字典，要求大模型基于 3D 物理常识和空间拓扑（支撑、包含关系）进行思维链（CoT）推理。
    *   `call_mllm_api(som_image, prompt)`: 发送拼贴图和提示词给 GPT-4o 或 Claude 3.5 Sonnet。
    *   `parse_mllm_json(response)`: 利用正则或 JSON parser 鲁棒提取输出，得到字典 `{"1": "椅背", "2": "椅腿"...}`。

### 【阶段 5】端到端主控脚本与语义回传
*   **`pipeline.py`** -> `Arbitr3DPipeline` 类:
    *   `__init__()`: 懒加载所有深度学习模型（避免 OOM）。
    *   `run(point_cloud)`: 串联上述 1 到 4 阶段。
    *   **语义无损回传 (Hard Binding)**: 拿到阶段 4 的 JSON 后，遍历 $M$ 个 `Entity3D`，直接将 JSON 中的类别名称作为 Label 写入对应 `point_indices` 的 3D 点云数组中。
    *   **返回**: 带有语义 Label 和 Instance ID 的最终 3D 点云。

## 四、 推荐的开发与测试里程碑 (Milestones)

为了保证开发的高效推进，建议将大象拆解，按以下顺序逐步攻克：

### 🏁 Milestone 1: 渲染与反投影基建 (Pipeline 骨架)
*   **任务**: 跑通 `module_render` 和 `module_3d_lift/projector.py`。
*   **验证方法**: 给定一个 3D 模型，渲染 20 张图。手动在其中一张图上用 OpenCV 画一个 2D mask，验证 `back_project_2d_to_3d` 能否在 3D 空间精准高亮出对应的点云。**（这是整个 Arbitr3D 能够成立的物理基石）**

### 🏁 Milestone 2: 攻克“硬隔离”核心算法 (解决核心痛点)
*   **任务**: 完善 `merger.py` 和 `conflict_resolver.py`。
*   **验证方法**: 输入两张视角不同但包含同一个部件（比如椅背）的 2D Mask，经过反投影后必定在 3D 空间产生重叠点和毛刺。运行图平滑和微观隔离逻辑，使用 Open3D 可视化验证是否得到了一个完美剥离的 3D 边界。

### 🏁 Milestone 3: 伪造数据，打通 MLLM 仲裁庭
*   **任务**: 实现 `som_renderer.py` 和 `mllm_arbitrator.py`。
*   **验证方法**: **无需等待前置模块完善**。用画图软件手工制作一张 Set-of-Mark 的拼贴图，手写一个 JSON 格式的先验字典，直接送给 GPT-4o API 进行测试，不断打磨 Prompt（特别是 CoT 的诱导词），确保其输出的 JSON 格式 100% 稳定且逻辑正确。

### 🏁 Milestone 4: 全链路串联与评估
*   **任务**: 引入真实的 SAM/GLIP (`module_2d_fm`)，完成 `pipeline.py`，并对齐 `evaluation/metrics.py`。
*   **验证方法**: 在 PartNetE 的小批量验证集上跑通全流程，计算 mIoU。重点排查大模型显存占用情况（适当清空 CUDA Cache）。
